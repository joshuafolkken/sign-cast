# リファクタリング支援プロンプト

このプロンプトは AI へのリファクタリング指示書です。

- **「リファクタリングを提案して」** → 分析・提案のみ行い、実装はユーザーに委ねる
- **「リファクタリングして」** → 優先度高・中を自動実施し、同一セッション内で収束するまで繰り返す

---

## 対象ファイルの特定（優先順位）

以下の優先順位で対象を決定する（上位が該当すれば下位は参照しない）：

1. **Staged Changes** および **Staged されていない Changes**（作業ツリーの変更）→ 対象の中心とする
2. **main 以外のブランチ** → そのブランチの変更内容（main との差分）も対象に含める
3. **上記が一切ない場合**（main ブランチで変更なし等）→ `src/` 配下のすべてのファイルを対象とする

上記で特定したファイルを起点に、**4.3 二段階スコープ拡張**で関連ファイルを追加する（`src/` 全体は対象にならない）。

### 除外条件

- `demo` フォルダ / ファイル名に `demo` を含むファイル
- `src/routes/stories` フォルダ
- ファイル先頭に `/* @refactor-ignore */` があるファイル

---

## 4.1 クロスファイルパターン検索（必須）

提案確定前に、対象スコープ全体で以下を検索する：

- **同一・類似の文字列リテラル**: 同じ Tailwind クラス、マジックナンバー、文言が他ファイルにないか
- **同一・類似の型定義**: `'desktop' | 'mobile'` など重複している型が他にないか
- **同一・類似の構造**: 同じ UI パターンが他にないか
- **定数化の余地**: ハードコードされた値が既存の定数ファイルと共通化できないか

---

## 4.2 実行必須の検索チェックリスト（漏れ防止）

候補確定前に必ず実行し、結果を記録する（「該当なし」の場合も明記）：

1. **行数・複雑度**: 300行超のファイル、`coding-standards.md` の複雑度閾値超の関数を検索
2. **名前空間オブジェクト**: 個別エクスポートを検索（定数のみのファイルは除外）
3. **型の安全性**: `any` の使用、`as` による型アサーションを検索。データ配列に明示的な型注釈があれば `as const satisfies T` への置き換えを検討
4. **重複文字列**: 同一・類似の文字列リテラルが2以上あるか
5. **重複コード**: 「同じ処理」「似たロジック」を検索
6. **マジックナンバー**: 数値リテラルを検索
7. **副作用の冪等性**: intersection observer・イベントハンドラ・コールバックなど、一度だけ実行すべき副作用に状態チェックのガード節があるか確認

---

## 4.3 二段階スコープ拡張

関連ファイルの拡張は2段階行う：

- **1段階目**: インポート元・インポート先、同ディレクトリ、同機能領域（例: sticky-header 関連なら constants, components, utils すべて）、類似名のファイル
- **2段階目**: 1段階目で得たファイルそれぞれに同様の拡張を実施（新規0件になるまで、最大3段階）

---

## 「リファクタリングを提案して」の動作

1. 「対象ファイルの特定」で対象を決め、**4.1・4.2・4.3** を実施してリファクタリング候補を網羅的に洗い出す
2. 高・中優先度の候補 → すべて提案（件数制限なし）／すべて低優先度の場合のみ → 最大3つまで提案
3. 提案のみ行い、実装はユーザーに委ねる
4. コメント提案は不要。実装可能なサンプルコードを提供する
5. このプロンプトの内容説明は不要。直接提案に移る

---

## 「リファクタリングして」の動作

**対象範囲**は「対象ファイルの特定」と同一（変更しない）。

```
[対象の特定] 上記「対象ファイルの特定」に従い決定
[1回目]    4.2 + 4.1 + 4.3 で網羅的分析 → 優先度高・中をすべて実施
[2回目以降] 同じ対象範囲内で再度網羅的分析 → 漏れがあれば実施
[収束判定]  優先度高・中が0件
[最終検証]  同じ対象範囲内で 4.2 を再実行 → 新規候補があれば実施してループへ
[完了]     最終検証でも0件 → 低優先度のみ報告して完了
```

- 優先度高・中が **0件になるまで**、同一セッション内で繰り返す（ユーザー再依頼不要）
- 低優先度は自動実施せず、ユーザーへの報告のみ
- 繰り返しのたびに「N回目の分析」「残り候補: X件」など進捗を簡潔に示す

---

## リファクタリング候補の優先度

コーディング規約・命名規則は `prompts/coding-standards.md` を参照。

### 高優先度

1. **メインファイルの分割**: 再利用可能・独立した責任を持つ機能を別ファイルに分離
2. **関数の分割**: 複雑度が `coding-standards.md` の基準を超える関数、複数責任を持つ関数
3. **ファイルの分割**: 300行超、複数責任、関連性の低い機能が混在
4. **名前空間オブジェクトエクスポートの適用**: 個別エクスポートを名前空間オブジェクト化（定数のみのファイルは除外）
5. **型定義の改善**: `any` の使用、型定義の不足。データ配列はリテラル型を保持するために `as const satisfies T` を使用しているか確認する（型注釈のみでは `string` に widened される）
6. **副作用関数の冪等性**: 同じ状態変化やコールバック呼び出しが複数回発生しないか確認する。特に intersection observer・イベントハンドラなど「一度だけ実行すれば十分な処理」は、状態チェックによる早期 return で保護する

### 中優先度

7. **定数の抽出**: マジックナンバー・繰り返し値の定数化、ハードコード文字列の整理
8. **変数名の改善**: 命名規則に合致しない名前、意味が不明確な名前の見直し
9. **コードクローンの排除**: 同一・類似コードが2箇所以上 → 必ず共通化を提案
10. **メソッド抽出**: 行数が多い・複数責任の関数を処理の塊ごとに抽出
11. **共通コンポーネントの作成**: 同じ UI・振る舞いが複数箇所にある場合、props で差を吸収する共通コンポーネントに統合

### 低優先度

12. **コードの整理**: 未使用の変数・関数の削除、不要なコメントの削除

---

## 基本方針

- **可読性・保守性をパフォーマンスより優先**する
- コードクローンの排除・メソッド抽出・共通コンポーネント化は必ず分析・提案に含める
- **提案しないパターン**: `async () => await fn()` → `() => fn()` への簡素化は行わない
- 提案が特にない場合は、無理に提案しない

---

## 提案の形式

各提案は以下の構成で記述する：

- **優先度**: 高 / 中 / 低
- **現在の状況**: 問題点
- **提案内容**: 具体的な改善案と期待される効果
- **実装方法**: ステップバイステップの手順
- **サンプルコード**: `coding-standards.md` の規約に準拠、lint エラーなし
- **関連ファイル**: 影響を受けるファイル

---

## 実装後の品質保証（絶対遵守）

リファクタリングを実装した場合、**例外なく以下を実行**する：

1. **`read_lints` ツールで全関与ファイルを確認**（省略禁止）
2. **エラーがあれば報告前に修正** → エラーが0件になるまで `read_lints` を繰り返す
3. **`pnpm run lint` でも検証**（`read_lints` が検出しないルールがあるため）
4. 完了報告時に「Lint エラーチェックを実施し、エラーがないことを確認しました」と明記する
5. **Lint エラーが残っている状態での完了報告は禁止**

## チェックリスト

- [ ] **「リファクタリングして」の場合**: 優先度高・中が0件になるまで同一セッション内で繰り返したか
- [ ] **「リファクタリングして」の場合**: 収束前に最終検証パス（4.2 再実行）を実施したか
- [ ] 4.3 二段階スコープ拡張を実施した
- [ ] 4.2 実行必須の検索チェックリスト（7項目）を実施し、結果を記録した
- [ ] 4.1 クロスファイルパターン検索を実施した
- [ ] サンプルコードに lint エラーがないことを確認した
- [ ] サンプルコードが `coding-standards.md` の規約に準拠している
- [ ] 提案件数のルールに従っている（高・中は制限なし、すべて低の場合のみ最大3つ）
- [ ] 提案のみの場合、実装はユーザーに委ねている
